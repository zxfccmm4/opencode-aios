# Product Principles

## Product First

A feature exists to help a user achieve an outcome.

Before building, understand:

- Who is this for?
- What problem does it solve?
- What does success look like?
- Can the solution be simpler?
- What should not be built?

## Reduce Cognitive Load

Prefer interfaces and flows that feel obvious.

Remove unnecessary decisions.

Avoid exposing implementation details to users.

## One Primary Action

Each screen or state should have one clear primary action.

Secondary actions should be visually and conceptually secondary.

## Empty, Loading, Error, and Success States

Every product surface should consider:

- Empty state
- Loading state
- Error state
- Success state
- Offline or degraded state, when relevant

## Shipping Judgment

Do not ship features that technically work but feel confusing, brittle, or unfinished.

A product should feel coherent, not merely functional.
